# E-Commerce Platform

(основано на реальных событиях)

## Предыстория

Вы продолжаете работать над платформой электронной коммерции `Igoroutine Shop`.

В предыдущей части вы уже реализовали сервисы:

- **Cart** — управление корзиной пользователя
- **Logistics and Order Management System (LOMS)** — логистика и управление заказами
- **Notifications** — отправка уведомлений о статусах заказа

На прошлом этапе `loms` асинхронно создавал уведомления через outbox, а сервис `notifications` доставлял их во внешний callback. Теперь нагрузка выросла, сервисы стали развиваться независимо, и прямой вызов `notifications` больше не подходит как основной способ интеграции.

# Часть 1

В этой домашней работе необходимо перевести доставку уведомлений о статусе заказа на брокер сообщений [Kafka](https://kafka.apache.org/).

Основной сценарий:

1. Пользователь создаёт заказ.
2. `loms` сохраняет заказ и outbox-сообщение в PostgreSQL.
3. Outbox worker публикует событие изменения статуса заказа в Kafka topic.
4. `notifications` читает событие из Kafka.
5. `notifications` отправляет callback во внешнюю систему по `CALLBACK_ADDR`.

Создание заказа не должно зависеть от скорости или доступности `notifications` и внешнего callback-сервиса.

Если в предыдущей домашней работе вы правильно выделили абстракции для отправки уведомлений, то эта часть должна решаться минимальными изменениями. Например, в `loms` может быть достаточно заменить реализацию `notificationClient`, который уже умеет `SendOrderStatusChangedNotification`, а в `notifications` добавить Kafka consumer (`RunConsumer`). Это не единственный возможный дизайн, но такая декомпозиция хорошо показывает, зачем нужны интерфейсы между бизнес-логикой и транспортом.

## Требования

Необходимо реализовать публикацию событий из `loms` в Kafka.

Событие должно содержать:

```json
{
  "user_id": 900001,
  "order_id": 123,
  "status": "awaiting_payment"
}
```

Необходимо реализовать чтение событий в `notifications` из Kafka.

Для доставки уведомлений ожидается гарантия `at least once`: если callback вернул ошибку или недоступен, consumer не должен считать сообщение успешно обработанным. После восстановления callback-сервиса это же событие должно быть обработано снова.

После успешной доставки callback не должен получать дубликаты одного и того же уведомления в штатном сценарии.

## Конфигурация

Kafka-параметры задаются через переменные окружения:

```yaml
KAFKA_BROKERS: "kafka-single:9092"
KAFKA_NOTIFICATIONS_TOPIC: "order_status_notifications"
KAFKA_CONSUMER_GROUP: "notifications"
```

Параметры outbox также задаются через переменные окружения:

```yaml
OUTBOX_WORKERS: "1"
OUTBOX_BATCH_SIZE: "10"
OUTBOX_FETCH_PERIOD: "200ms"
OUTBOX_IN_PROGRESS_TTL: "1s"
```

Kafka topic может создаваться автоматически при первой публикации.

## Тестирование

Решение будет проверяться интеграционными тестами курса.

Для проверки корректности Kafka-интеграции используются тесты в [notifications_test.go](./integration-tests/notifications_test.go). Они проверяют, что:

- создание заказа успешно завершается даже при ошибке callback;
- событие о статусе заказа реально появляется в Kafka;
- события публикуются для изменений статуса `awaiting_payment`, `paid` и `cancelled`;
- `notifications` доставляет сообщение минимум один раз после временной ошибки callback;
- после успешной доставки уведомление не отправляется повторно.

Тесты напрямую подключаются к Kafka и читают topic `order_status_notifications`, поэтому простая отправка уведомления по старому gRPC-пути не пройдёт проверку.

# Бонусная часть

За дополнительные баллы можно реализовать интеграцию `notifications` с отправкой сообщений в Telegram.

Для сдачи бонусной части необходимо записать видео до двух минут с демонстрацией работы интеграции, загрузить его на Яндекс Диск и заполнить бонусную форму. Не забудьте сделать видео доступным по ссылке.

В видео должно быть видно, что при изменении статуса заказа сообщение доходит до Telegram.

## Локальный запуск

- `task update hw=hw2` — скачивает docker-compose файлы для тестирования и локального запуска
- `task frontend` — запускает frontend, перед выполнением необходимо установить [npm](https://docs.npmjs.com/downloading-and-installing-node-js-and-npm)
- `task backend` — запускает backend
- На Windows без WSL многое может не работать

## Особенности реализации

* **Настоятельно** рекомендуется писать комментарии для проверяющих, чтобы обосновать свою точку зрения в выборе того или иного компромисса (trade-off’а).
* Крайне рекомендуется осознать механизм работы интеграционных тестов.
* При работе с Kafka важно явно продумать момент commit offset: сообщение можно коммитить только после успешной обработки.
* Не храните токены Telegram-бота и другие секреты в репозитории.

## Сдача

* Открыть pull request из ветки `hw2` в ветку `main` **вашего репозитория**, название ветки **важно**.
* В описании PR заполнить количество часов, которые вы потратили на это задание.
* Не стоит изменять файлы в директории [.github](.github).
* Для бонусной части заполнить бонусную форму и приложить ссылку на видео.

## Скрипты

Для запуска скриптов на курсе необходимо установить [go-task](https://taskfile.dev/docs/installation)

`go install github.com/go-task/task/v3/cmd/task@latest`

Перед выполнением задания не забудьте выполнить:

```bash
task update hw=hw2
```

При возникновении ошибок с `task update` ознакомьтесь с [гайдом](https://github.com/igoroutine-courses/guide.github_scripts)

Запустить линтер:
```bash
task lint
```

Запустить тесты:
```bash
task test
```

Обновить файлы задания:
```bash
task update hw=hw2
```

Принудительно обновить файлы задания:
```bash
task force-update hw=hw2
```

Запустить генерацию кода:
```bash
task generate
```

Запустить backend в docker:
```bash
task backend
```

Запустить frontend в docker:
```bash
task frontend
```

Скрипты работают на Windows, однако при разработке на этой операционной системе рекомендуется использовать [WSL](https://learn.microsoft.com/en-us/windows/wsl/install)
